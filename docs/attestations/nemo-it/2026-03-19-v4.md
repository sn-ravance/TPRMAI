## Findings

### High Findings

#### H1: OIDC State Parameter Not Validated in Callback

**What:** Your login system generates a security token (called "state") when it sends users to the login provider, but it never checks that token when the user comes back. This is like giving someone a claim ticket at coat check but never verifying it when they pick up a coat -- anyone could substitute their own ticket.

**Where:** `src/app/api/auth/callback/route.ts` (entire file -- state param never read); `src/app/api/auth/login/route.ts:13` (state generated but not stored for later verification)

**How:** Code review identified that the login route generates a cryptographic state parameter and includes it in the OIDC authorization URL, but the callback route never reads or validates the state parameter from the callback query string.

**Root cause:** The state parameter generation was implemented but the verification step was not completed. The state is generated and discarded.

**Risk matrix:** Likelihood: 3 (known technique) x Impact: 4 (session hijacking via login CSRF) = **12 (HIGH)**

**OWASP:** A07:2021 -- Identification and Authentication Failures

**Remediation:**
- Part A: In the login route, store the generated state in a short-lived HttpOnly cookie. In the callback route, read the state from both the query string and the cookie, compare them, and reject the request if they don't match.
- Part B: The OIDC provider (mock-oidc in dev, real IdP in prod) will echo back the state parameter. This is the standard CSRF protection mechanism for OAuth 2.0/OIDC flows per RFC 6749 Section 10.12.

---

#### H2: Missing Authorization on Orchestrator PUT and PATCH Endpoints

**What:** Two of your AI orchestration endpoints -- one that processes documents and one that runs maintenance -- don't check whether the logged-in user has permission to use them. Any authenticated user could trigger AI document processing or maintenance cycles, regardless of their role.

**Where:** `src/app/api/orchestrator/route.ts:89` (PUT handler -- no requirePermission call); `src/app/api/orchestrator/route.ts:133` (PATCH handler -- no requirePermission call)

**How:** Code review compared the POST handler (which correctly calls `requirePermission('agents', 'create')`) against the PUT and PATCH handlers, which have no authorization checks.

**Root cause:** The POST handler was implemented with RBAC, but PUT and PATCH were added later without the same authorization pattern.

**Risk matrix:** Likelihood: 3 (any authenticated user) x Impact: 3 (unauthorized AI operations, resource consumption) = **9 (HIGH)**

**OWASP:** A01:2021 -- Broken Access Control

**Remediation:**
- Part A: Add `const denied = await requirePermission('agents', 'create'); if (denied) return denied;` at the start of both PUT and PATCH handlers, matching the POST handler pattern.

### Medium Findings

#### M1: Stored XSS -- Script Tags Accepted in Vendor Name

**What:** Your vendor creation form accepts HTML and JavaScript code in the vendor name field without sanitizing it. An attacker could store a `<script>` tag as a vendor name, which could execute malicious code when other users view that vendor.

**Where:** `src/app/api/vendors/route.ts` (POST handler -- no HTML sanitization on input)

**How:** Dynamic testing submitted `<script>alert(1)</script>` as a vendor name via the API. The server accepted and stored it (HTTP 201). React's default JSX escaping prevents execution in the current frontend, but the data is stored unsanitized in the database, creating risk if rendered in any non-React context (emails, reports, exports).

**Root cause:** Zod validation checks field types and required fields but does not strip HTML/script tags from string inputs.

**Risk matrix:** Likelihood: 3 (easy to submit) x Impact: 3 (stored XSS if rendered unsafely) = **9 (MEDIUM)** -- mitigated to medium because React auto-escapes output, but the stored payload remains a risk.

**OWASP:** A03:2021 -- Injection

**Remediation:**
- Part A: Add an HTML sanitization step (strip `<script>`, `<iframe>`, event handlers) to all user-facing text inputs before database storage, either in Zod schemas or as a middleware.
- Part B: Continue relying on React's auto-escaping for frontend rendering. Ensure any report generation (DOCX, PDF) also escapes stored content.

#### M2: No Input Length Validation on Text Fields

**What:** Your API accepts extremely long text in fields like vendor name (tested with 10,000 characters). This could cause display issues, performance problems, or be used to exploit buffer handling in downstream systems.

**Where:** `src/app/api/vendors/route.ts` (POST handler -- Zod schema lacks `.max()` on string fields)

**How:** Dynamic testing submitted a 10,000-character vendor name. The server accepted and stored it (HTTP 201).

**Root cause:** Zod schemas define required fields and types but don't set maximum lengths.

**Risk matrix:** Likelihood: 3 x Impact: 2 = **6 (MEDIUM)**

**OWASP:** A05:2021 -- Security Misconfiguration

**Remediation:**
- Part A: Add `.max(255)` (or appropriate length) to all string fields in Zod validation schemas. Common limits: name fields 255 chars, description/summary 5000 chars, notes 10000 chars.

---

#### M3: NODE_TLS_REJECT_UNAUTHORIZED=0 in Dockerfile Build Stages

**What:** During the Docker image build process, TLS certificate verification is temporarily turned off while downloading dependencies. This is like accepting packages from a delivery person without checking their ID -- a sophisticated attacker on the network could substitute malicious packages.

**Where:** `Dockerfile:13` and `Dockerfile:24` (re-enabled at lines 16 and 26)

**How:** Static analysis of Dockerfile. The comment references Zscaler SSL-inspecting proxy as the reason.

**Root cause:** Corporate SSL-inspecting proxy (Zscaler) breaks TLS verification during npm install. Rather than injecting the corporate CA certificate, TLS verification was disabled as a workaround.

**Risk matrix:** Likelihood: 2 (requires MITM during build) x Impact: 4 (supply chain compromise) = **8 (MEDIUM)**

**OWASP:** A08:2021 -- Software and Data Integrity Failures

**Remediation:**
- Part A: Inject the corporate CA certificate into the Docker build image: `COPY corporate-ca.crt /usr/local/share/ca-certificates/ && update-ca-certificates` and set `NODE_EXTRA_CA_CERTS=/usr/local/share/ca-certificates/corporate-ca.crt`.
- Part B: If CA cert injection is not feasible, document this as an accepted risk with a comment explaining the threat model and the compensating control (Zscaler itself verifies the upstream certificates).

#### M4: Docker Base Image Not Fully Pinned

**What:** Your Docker build uses `node:20-alpine` which is a floating tag. Different builds at different times may pull different versions, making builds non-reproducible.

**Where:** `Dockerfile:1`

**How:** Dockerfile analysis.

**Root cause:** Standard practice to use major-version tags for convenience, but it introduces reproducibility risk.

**Risk matrix:** Likelihood: 2 x Impact: 3 = **6 (MEDIUM)**

**OWASP:** A05:2021 -- Security Misconfiguration

**Remediation:**
- Part A: Pin to a specific digest: `node:20-alpine@sha256:<digest>` or at minimum a minor version: `node:20.x.y-alpine`.

### Low Findings

#### L1: Malformed JSON Returns 500 Instead of 400

**What:** When someone sends badly-formatted data to your API, the server returns a "500 Internal Server Error" instead of a "400 Bad Request." This doesn't expose sensitive information, but it masks the true nature of the error and could confuse monitoring systems.

**Where:** `src/app/api/vendors/route.ts` (POST handler)

**How:** Dynamic testing sent `{invalid}` as JSON body. Server returned HTTP 500 with `{"error":"Failed to create vendor"}`.

**Root cause:** The try/catch block catches JSON parse errors but treats them as internal errors rather than client input errors.

**Risk matrix:** Likelihood: 3 x Impact: 1 = **3 (LOW)**

**OWASP:** A05:2021 -- Security Misconfiguration

**Remediation:**
- Part A: Add explicit JSON parse error handling that returns 400 before the Zod validation step.

---

#### L2: No General API Rate Limiting on Non-AI Endpoints

**What:** Your regular API endpoints (vendors, documents, etc.) don't have rate limiting. Someone could send thousands of requests per second to overload your server. Your AI endpoints are properly rate-limited, but the standard CRUD endpoints are not.

**Where:** All API routes under `src/app/api/` except AI agent routes

**How:** Dynamic testing sent 30 rapid requests to `/api/vendors`. All returned HTTP 200 with no throttling.

**Root cause:** Rate limiting was implemented for AI endpoints (via `aiRateLimit` middleware) but not for general API endpoints.

**Risk matrix:** Likelihood: 3 x Impact: 2 = **6 (LOW)** -- reduced because the app is behind OIDC auth, limiting the attack surface to authenticated users.

**OWASP:** A04:2021 -- Insecure Design

**Remediation:**
- Part A: Add general rate limiting middleware (e.g., 100 requests/minute per user) to all API routes. The existing `AI_RATE_LIMIT` pattern can be adapted.

---

#### L3: JWT Secret and OIDC Secret Fall Back to Known Defaults (Non-Fatal)

**What:** If someone deploys your app without setting the JWT secret or OIDC secret, the app logs a warning but continues running with well-known default values. An attacker who knows these defaults could forge authentication tokens.

**Where:** `src/lib/auth.ts:9-12`, `src/middleware.ts:4-7`, `src/lib/oidc.ts:9-11`

**How:** Code review. All three files log `console.error` in production but do not halt the process.

**Root cause:** Defensive approach to avoid crashes, but in production the consequence of running with default secrets is worse than crashing.

**Risk matrix:** Likelihood: 2 (requires misconfigured deployment) x Impact: 4 (full auth bypass) = **8** but rated **LOW** because docker-compose.yml already sets non-default values, and the warning provides defense-in-depth.

**OWASP:** A02:2021 -- Cryptographic Failures

**Remediation:**
- Part A: Change `console.error` to `throw new Error()` or `process.exit(1)` in production for both JWT_SECRET and OIDC_CLIENT_SECRET, ensuring the app refuses to start with known-insecure defaults.

### Informational Findings

#### I1: Zod Validation Errors Expose Schema Structure

**What:** When input validation fails, the API returns detailed Zod error objects that reveal field names, expected types, and validation constraints. This is helpful for legitimate API consumers but gives attackers a map of your data model.

**Where:** 11 API routes return `{ error: 'Validation failed', details: error.errors }`

**How:** Static analysis of error handling patterns.

**OWASP:** A04:2021 -- Insecure Design

**Remediation:** Consider returning only field names and generic messages (e.g., "name is required") rather than the full Zod error object in production.

---

#### I2: AI Output Validation Rules Defined But Never Called

**What:** Your AI safety module includes a function called `validateAgentOutput()` with comprehensive validation rules for risk scores, assessment scores, findings, and reports. However, this function is never actually called anywhere in the codebase -- it's dead code. Only the JSON parsing function (`safeParseJSON`) is used.

**Where:** `src/lib/ai/validate.ts:27` (function defined); not called in `base-agent.ts` or any route

**How:** AI safety control verification -- grep for function usage returned zero matches outside the definition file.

**Risk matrix:** No direct security impact, but AI responses could contain out-of-range scores or hallucinated enum values without detection.

**Remediation:** Wire `validateAgentOutput()` into the base agent pipeline after `safeParseJSON()`, passing the appropriate rule set based on agent type.

---

#### I3: Conversation History Limits Declared But Not Implemented

**What:** Your `.env.example` declares `AI_MAX_HISTORY_TURNS=20` as a configurable limit, but no code references this variable. The current architecture is single-turn (one request, one response), so this is not an active risk.

**Where:** `.env.example:64` (variable declared); zero references in `src/`

**How:** AI safety control verification.

**Remediation:** Either implement history truncation when multi-turn conversations are added, or remove the variable from `.env.example` to avoid documenting a control that doesn't exist.

### Current Findings (v3)

#### M1: Content-Type Middleware Blocks Empty-Body POST Requests

**What:** The Content-Type validation middleware requires a content type on all POST requests to non-public paths. Some POST endpoints like `/api/auth/logout` send no body and therefore no Content-Type header. This causes the middleware to return 415 (Unsupported Media Type) instead of processing the request.

**Where:** `src/middleware.ts`, lines 81-92

**How:** Dynamic testing showed `POST /api/auth/logout` without a Content-Type header returns 415 instead of clearing the JWT cookie.

**Root Cause:** The Content-Type check applies to all POST/PUT/PATCH requests on non-public paths. The logout endpoint is not in the PUBLIC_PATHS list, and an empty-body POST has no Content-Type header.

**Risk Matrix:** Likelihood: 4 x Impact: 2 = **8 (MEDIUM)**
An attacker cannot exploit this, but it prevents legitimate logout from working via a simple POST without headers.

**OWASP:** A05:2021 - Security Misconfiguration

**Remediation:**
- Add `/api/auth/logout` to the Content-Type exemption list, OR
- Allow POST requests with no Content-Type when the body is empty (content-length: 0)
- Ensure the frontend logout button sends `Content-Type: application/json` header

#### M2: Hardcoded JWT Secret Fallback

**What:** The JWT signing secret has a hardcoded fallback value (`'dev-secret-change-in-production'`) used when the `JWT_SECRET` environment variable is not set. If this fallback is accidentally used in production, anyone who knows the fallback string could forge valid JWTs.

**Where:** `src/middleware.ts` line 5, `src/lib/auth.ts` line 10

**How:** Static analysis found `process.env.JWT_SECRET || 'dev-secret-change-in-production'` pattern.

**Root Cause:** Defensive coding to allow the app to start without config. The fallback is clearly named but still represents a risk if .env is misconfigured.

**Risk Matrix:** Likelihood: 2 x Impact: 5 = **10 (MEDIUM)**
Requires a production deployment to miss the JWT_SECRET variable. Docker Compose sets it explicitly, reducing likelihood.

**OWASP:** A02:2021 - Cryptographic Failures

**Remediation:**
- Throw an error at startup if `JWT_SECRET` is not set and `NODE_ENV === 'production'`
- Keep the fallback for development only
- Add a startup check that logs a warning if using the default secret

#### M3: OIDC Subject Mismatch Between Seed Data and Mock-OIDC

**What:** The database seed script creates users with OIDC subjects like `admin-001`, `analyst-001`, but the mock-OIDC service sends subjects like `mock-admin`, `mock-analyst`. When a mock-OIDC user logs in, the auth callback cannot find a matching user and creates a new one with the default VIEWER role.

**Where:** `prisma/seed.ts` (oidcSubject values) vs `mock-services/mock-oidc/app.py` (SEED_USERS)

**How:** Dynamic testing: logging in as `mock-admin` via OIDC results in a VIEWER role instead of ADMIN.

**Root Cause:** The seed data and mock-OIDC service were created independently with different subject identifier conventions.

**Risk Matrix:** Likelihood: 5 x Impact: 2 = **10 (MEDIUM)**
This is a development-environment issue only. Every mock-OIDC login gets the wrong role, preventing proper testing of RBAC.

**OWASP:** A07:2021 - Identification and Authentication Failures

**Remediation:**
- Align the seed data `oidcSubject` values with mock-OIDC `sub` values (e.g., change seed to use `mock-admin`, `mock-analyst`, `mock-viewer`, `mock-user`)
- OR update mock-OIDC SEED_USERS to use `admin-001`, `analyst-001`, etc.

#### L1: Hardcoded Development Credentials in docker-compose.yml

**What:** Docker Compose contains hardcoded database passwords and JWT secrets directly in the file rather than referencing .env variables.

**Where:** `docker-compose.yml`, lines 13, 19-20, 67-68

**How:** Static analysis found `POSTGRES_PASSWORD=tprm_pass`, `JWT_SECRET=docker-dev-jwt-secret-not-for-production`.

**Root Cause:** Simplified local development setup. Values are clearly marked as dev-only.

**Risk Matrix:** Likelihood: 1 x Impact: 2 = **2 (LOW)**
Values are clearly labeled "not-for-production" and only used in local Docker development.

**OWASP:** A05:2021 - Security Misconfiguration

**Remediation:**
- Use `${VARIABLE:-default}` syntax in docker-compose.yml to reference .env values with fallbacks
- Document that production deployments must use a secrets manager

---

#### L2: Hardcoded OIDC Client Secret Fallback

**What:** The OIDC client configuration has a hardcoded fallback of `'mock-oidc-secret'` when the environment variable is not set.

**Where:** `src/lib/oidc.ts`, client_secret assignment

**How:** Static analysis grep for hardcoded secret patterns.

**Root Cause:** Same defensive coding pattern as JWT secret -- allows the app to work with mock-OIDC without explicit configuration.

**Risk Matrix:** Likelihood: 1 x Impact: 3 = **3 (LOW)**
The fallback only matches the mock-OIDC service. A real OIDC provider would reject this secret.

**OWASP:** A07:2021 - Identification and Authentication Failures

**Remediation:**
- Add startup validation: if `NODE_ENV === 'production'` and OIDC_CLIENT_SECRET is unset or equals the mock value, throw an error

#### I1: Windows File Paths in Seed Data

**What:** The database seed script contains hardcoded Windows file paths with a username and company directory (OneDrive path) visible in the seed data.

**Where:** `prisma/seed.js`, lines 256, 272

**How:** Static analysis found `C:\\Users\\JOBLER\\OneDrive - Sleep Number Corporation\\...`

**Root Cause:** Seed data was created from actual development environment paths.

**Risk Matrix:** Likelihood: 1 x Impact: 1 = **1 (INFORMATIONAL)**
Leaks a username and company name in the seed database, but this is development data only.

**OWASP:** A04:2021 - Insecure Design

**Remediation:**
- Replace with relative paths or generic placeholder paths in seed data

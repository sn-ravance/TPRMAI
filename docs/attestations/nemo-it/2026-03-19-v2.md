## Findings

### High Findings

#### H1: Next.js 16.1.6 has 5 known security advisories

**What:** The web framework your app is built on has 5 published security issues. The most serious one allows an attacker to bypass the cross-site request forgery (CSRF) protection on Server Actions, meaning they could trick a logged-in user's browser into performing actions without their knowledge.

**Where:** `package.json` -- `next@16.1.6`

**How:** Detected by `npm audit`. Advisory IDs: GHSA-mq59-m269-xvcx (CSRF bypass), GHSA-jcc7-9wpm-mj36 (HMR CSRF), GHSA-3x4c-7xq6-9pq8 (image cache DoS), GHSA-h27x-g6w4-24gq (postponed buffer DoS), GHSA-ggv3-7p47-pfv8 (HTTP request smuggling).

**Root cause:** The installed version (16.1.6) predates the security patches released in 16.1.7.

**Risk matrix:**
- Likelihood: 3 (known technique, publicly documented)
- Impact: 4 (CSRF can trigger state-changing actions as an authenticated user)
- **Risk Score: 12 (HIGH)**

**Remediation:**
- Programmatic fix: Run `npm install next@16.1.7` (or latest 16.x)
- Compensating control: The OIDC + JWT auth model with SameSite cookies already mitigates some CSRF vectors

**OWASP:** A06:2021 -- Vulnerable and Outdated Components

---

#### H2: flatted 3.4.1 -- Prototype Pollution via parse()

**What:** A library used for JSON serialization has a prototype pollution vulnerability. An attacker who can control data passed to `flatted.parse()` could inject properties into all JavaScript objects, potentially bypassing security checks or causing unexpected behavior.

**Where:** `node_modules/flatted` (dependency of `flat-cache`, used by `eslint`)

**How:** Detected by `npm audit`. Advisory: GHSA-rf6f-7fwh-wjgh (CWE-1321).

**Root cause:** The `parse()` function in flatted <= 3.4.1 does not properly sanitize `__proto__` keys during deserialization.

**Risk matrix:**
- Likelihood: 2 (requires attacker-controlled input reaching flatted.parse)
- Impact: 4 (prototype pollution can escalate to RCE in some contexts)
- **Risk Score: 8 (MEDIUM-HIGH)**

**Remediation:**
- Programmatic fix: `npm update flatted` to >= 3.4.2
- Note: This is a transitive dependency through eslint's flat-cache; the direct application code does not call flatted.parse() with user input, which significantly reduces exploitability

**OWASP:** A08:2021 -- Software and Data Integrity Failures

#### H4: No general API rate limiting on non-AI endpoints

**What:** Your AI-related endpoints have rate limiting (good!), but the rest of your API does not. An attacker could flood your regular API endpoints (vendors, assessments, findings, etc.) with thousands of requests per second, potentially making the app unavailable to real users or exhausting database resources.

**Where:** All API routes under `src/app/api/` except the agent routes and orchestrator

**How:** Sent 50 rapid requests to `/api/vendors` -- all returned 200 with no throttling or 429 responses.

**Root cause:** The `aiRateLimit` middleware from `src/lib/ai/rate-limit.ts` is only applied to AI agent routes. Standard CRUD routes have no rate limiting.

**Risk matrix:**
- Likelihood: 4 (trivially automated with any HTTP tool)
- Impact: 3 (service degradation, database exhaustion)
- **Risk Score: 12 (HIGH)**

**Remediation:**
- Programmatic fix: Add a general-purpose rate limiter middleware to all API routes (e.g., Next.js middleware.ts with a sliding window per IP/user)
- Technological control: Deploy a reverse proxy (nginx, Cloudflare) with rate limiting rules in front of the app
- Compensating control: The OIDC authentication requirement already limits unauthenticated abuse

**OWASP:** A05:2021 -- Security Misconfiguration

#### H5: Content-Security-Policy uses unsafe-inline and unsafe-eval

**What:** The security header that controls which scripts can run in the browser currently allows inline scripts and eval(). This weakens protection against cross-site scripting (XSS) attacks -- if an attacker finds any way to inject HTML, they could execute JavaScript in users' browsers.

**Where:** Security headers middleware (CSP header configuration)

**How:** Inspected `Content-Security-Policy` response header. Found `script-src 'self' 'unsafe-inline' 'unsafe-eval'`.

**Root cause:** Next.js requires `unsafe-inline` for its hydration scripts in development/production. `unsafe-eval` may be needed by some dependencies but should be investigated for removal.

**Risk matrix:**
- Likelihood: 2 (requires a pre-existing injection point)
- Impact: 4 (full XSS exploitation if injection found)
- **Risk Score: 8 (MEDIUM-HIGH)**

**Remediation:**
- Programmatic fix: Use Next.js nonce-based CSP (`script-src 'nonce-{random}'`) to replace `unsafe-inline`. Investigate whether `unsafe-eval` can be removed (test thoroughly).
- Compensating control: The application uses React's built-in XSS protection (no `dangerouslySetInnerHTML` found in app code), which significantly reduces the practical risk

**OWASP:** A05:2021 -- Security Misconfiguration

### Medium Findings

#### M1: X-Powered-By header discloses technology stack

**What:** The server tells every visitor what technology it runs on. While not dangerous on its own, this gives attackers a head start -- they know exactly which framework-specific attacks to try.

**Where:** HTTP response headers -- `X-Powered-By: Next.js`

**Risk matrix:** Likelihood: 5 x Impact: 1 = **5 (MEDIUM)**

**Remediation:** Add `poweredByHeader: false` to `next.config.js`, or set `X-Powered-By` to a misleading value in the security headers middleware.

**OWASP:** A05:2021 -- Security Misconfiguration

---

#### M2: JWT tokens not invalidated on logout (stateless design)

**What:** When a user logs out, the app clears the cookie from their browser, but the JWT token itself remains technically valid until it expires. If someone copied the token before logout, they could use it until expiry.

**Where:** `src/app/api/auth/logout/route.ts` -- clears cookie but no server-side token revocation

**Risk matrix:** Likelihood: 2 x Impact: 3 = **6 (MEDIUM)**

**Remediation:**
- Programmatic fix: Implement a short-lived token blocklist in Redis or database (check on each request)
- Compensating control: Keep JWT expiry short (e.g., 15 minutes) and use refresh token rotation. The current OIDC + SameSite cookie approach already limits token theft vectors.

**OWASP:** A04:2021 -- Insecure Design

---

#### M3: Missing Secure flag on JWT cookie in development

**What:** The authentication cookie doesn't have the `Secure` flag set when running over HTTP (localhost development). This is expected for local development but must be enforced in production.

**Where:** JWT cookie configuration in auth routes

**Risk matrix:** Likelihood: 1 (dev only) x Impact: 3 = **3 (LOW in dev, MEDIUM if deployed without fix)**

**Remediation:** Verify the cookie sets `Secure: true` when `NODE_ENV=production` or when served over HTTPS. The current implementation derives this from the URL protocol, which is correct behavior.

**OWASP:** A05:2021 -- Security Misconfiguration

---

#### M4: CORS allows credentials with broad origin in development

**What:** The CORS configuration may allow credentialed requests from origins that should not have access. In development this is typically fine, but production must restrict origins.

**Where:** Next.js API route CORS headers

**Risk matrix:** Likelihood: 2 x Impact: 3 = **6 (MEDIUM)**

**Remediation:** Ensure production CORS configuration uses an explicit allowlist of trusted origins rather than wildcard or overly broad patterns.

**OWASP:** A01:2021 -- Broken Access Control

---

#### M5: PII masker module-level counter is not thread-safe

**What:** The PII masking module uses a module-level counter (`let counter = 0`) to generate unique placeholders. In a serverless or multi-worker environment, this counter could produce duplicate placeholder IDs, causing PII restoration to map incorrectly.

**Where:** `src/lib/ai/pii-masker.ts:20-23` -- global `counter` variable with `nextId()` function

**Risk matrix:** Likelihood: 2 x Impact: 3 = **6 (MEDIUM)**

**Remediation:** Use `crypto.randomUUID()` or a request-scoped counter instead of a module-level incrementing counter. In the current single-process Docker deployment, this is low risk, but would become problematic under horizontal scaling.

**OWASP:** A04:2021 -- Insecure Design

---

#### M6: Error responses on 404 routes may reveal framework information

**What:** Requesting non-existent API routes returns Next.js default error pages that identify the framework version. This is similar to the X-Powered-By finding but applies to error pages.

**Where:** Default Next.js 404/500 error handlers

**Risk matrix:** Likelihood: 4 x Impact: 1 = **4 (MEDIUM)**

**Remediation:** Add custom error pages (`app/not-found.tsx`, `app/error.tsx`) that do not reveal framework details.

**OWASP:** A05:2021 -- Security Misconfiguration

---

#### M7: Docker Compose uses plain-text database credentials

**What:** The PostgreSQL username, password, and database name are hardcoded in `docker-compose.yml`. While this is acceptable for local development, these credentials must not be used in production.

**Where:** `docker-compose.yml:13,43-44` -- `tprm_user` / `tprm_pass`

**Risk matrix:** Likelihood: 2 x Impact: 3 = **6 (MEDIUM)**

**Remediation:** For production, use Docker secrets, environment variable injection from a secrets manager, or a .env file that is not committed to version control. The current setup is appropriate for local dev only.

**OWASP:** A05:2021 -- Security Misconfiguration
